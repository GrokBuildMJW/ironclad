# Project status & wiring

> ## ⚠️ Read this first
>
> Ironclad's engine grew out of a **proven, in-production orchestrator** (years of
> daily use). It is currently undergoing a **complete redesign**: the original
> single-process CLI is being split into a headless **server** + thin **client**,
> containerized, and given a reasoning-worker fan-out and a purpose-built TypeScript terminal client.
>
> **Not everything is re-wired or re-tested yet.** The pieces below are marked
> honestly: *proven* (inherited, battle-tested), *wired + tested* (rebuilt and
> verified in the new architecture), *placeholder* (hook exists, backend/logic not
> yet ported), or *opt-in* (off by default). Treat `main` as a development snapshot.

This document is the single source of truth for **what actually works right now**.

> **Release model.** Ironclad is **pre-release** (`0.0.x`, alpha). Tagged releases are
> published on **PyPI** as `ironclad-ai` and as **GitHub Releases** (latest `v0.0.21`);
> the importable wheel is the Agent-Contract-Kernel (`ack` + `ack.lodestar`), while the
> orchestration engine ships as runnable scripts until its API stabilizes. There is **no
> stable/1.0 release yet** — APIs, layout, and config may change between `0.0.x` versions.

## Module reference

| File | What it does |
|------|--------------|
| `engine/gx10.py` | The orchestrator **engine library** (the *proven* core). Agent loop, tool execution, deterministic `TaskStore`, fail-closed macros (`stage_handover` / `advance_pipeline`), config-tree loader, platform detection, context trimming. The server imports it; the standalone monolithic CLI was **removed** (one way: server + client). |
| `engine/server.py` | **Headless server** (new). Drives the engine with no UI, exposes plain-HTTP endpoints, serializes agent access, runs the feedback-side reconciler. |
| `clients/ink/` | **Recommended terminal client** (new, TypeScript). Purpose-built renderer (own React reconciler + flexbox + packed cell buffer + cell-level diff) → **ghost-free resize**, smooth streaming, native scrollback/selection/copy. Slash-command autocomplete, local `!cmd` shell, in-CLI `/update`, per-project session. Same HTTP/tool-bridge contract; build-from-source (Node ≥ 22). |
| `engine/client.py` | **Thin client (legacy)**. Line REPL + the local code-agent pool (`claude --print`) that keeps project code on your machine. Zero-dependency fallback. |
| `engine/tui.py` | **Full-screen client (legacy)**. prompt_toolkit UI, live streaming, status toolbar, scrollback, compressed multi-line paste. |
| `engine/commands.py` | Shared command router for REPL + TUI (`/command` → local or forwarded to the server). |
| `engine/workers.py` | **Reasoning-worker fan-out** (new). Independent prompts run concurrently against the model (`POST /fanout`). |
| `ack/` | **Agent-Contract-Kernel**: schema-as-SSOT (`case_spec`), bounded validate→reask (`validated_emit`), constrained emission (`constrained_emission`), the shared failure-classification taxonomy (`failure_class`), the failure→action policy (`strategy`), per-TaskType loop budgets (`loop_profile`), the mark-only evaluation layer (`verify`), the output-quality circuit breaker (`quality`), the process-self-correction policy (`process`), the Loop-Intelligence Hook-Bus (`hooks`), registry, doctor, generator, the opt-in `lodestar` plugin. |

| **Project isolation** (`/project`, `/switch`, installation-global registry) | **wired + tested** | Every engine session runs in a *project*. An installation-global **registry** (under `GX10_HOME`, atomic + OS-file-locked) is the SSOT of registered projects + the persisted continuity pointer; the engine's current project is **single-active per process** (cached, never re-read per request, so a second engine can't yank a running one). `/project list\|new <name> [--type] [--path]\|active\|track …` manages projects (the guided setup command, #601 S16: `new` mints a fresh isolated project — root `<cwd>/<slug>` or `--path`, minted `mem_ns`, made active, and `--type` seeds the first vault unit; `delete <id> [--purge]` forgets the project's memory + drops the registry entry [dirs kept unless `--purge`, which is guarded against the cwd/boot/home]; `archive`/`unarchive` toggle a reversible hidden flag [data kept, not switchable, hidden from `list` unless `--all`]; `/initiative` is a deprecated alias); `/switch <project_id>` quiesces and rebinds (and refuses an archived target) — saving the leaving conversation under its own root, binding the target's paths (`state_root`/`vault_root` resolve under the project root) + its own **memory partition** (`mem_ns`), and swapping the conversation (no cross-project bleed). The implicit **`default`** project binds the boot workdir + the legacy/base memory partition, so a single-project install is byte-identical to before. A **registered** project is **fail-closed** at bind (#601 S14-2): a non-`default` project with an invalid/empty `mem_ns` is refused (it would otherwise silently leak into the base partition); boot degrades to the safe `default`, a `/switch` rolls back. The memory handle is a single installation-global connection re-keyed **per call** by the active partition (no rebuild-on-switch). A switch is **refused** while a dev unit is in-flight for either side; the system prompt is project-independent and preserved. The model's **file tools, `execute_command`, and a launched code-agent run with the active project's root as their working directory** (the switch does not `chdir` the process — the exec cwd is threaded through the active context; absolute paths verbatim; `default` = the boot workdir, byte-identical). A project can carry parallel **tracks** (single-active; [ADR-0011](adr/0011-dev-process-rework-project-isolation.md) AD-2', #601 S12a): a non-`main` active track gets its own isolated vault subtree under `.tracks/<track>/` (`vault_root` is track-aware), while the default `main` track is byte-identical to the pre-track layout. Tracks are managed from the CLI with `/project track new|use|list` (#601 S16): `new` creates-and-switches, `use` switches, `list` shows them — a switch rebinds the context so the vault subtree + memory sub-scope + per-track library follow. Vault mutation is **serialized** per project+track (#601 S12b): a `vault_lock(pid, track)` distinct from the dev-loop `project_lock` wraps the vault writers (`initiative_new`/`reconcile_vault` + the macros `_stage_handover`/`_advance_pipeline`) — reentrant within a call stack, OS-serialized cross-process, fail-soft. See [`project-isolation.md`](project-isolation.md). |

Dependency-inverted (`set_driver`/`get_driver` + a `runtime_checkable` `DevProcessDriver` protocol): the facade imports nothing from the engine, ships in the wheel (`ack.devprocess`), and degrades cleanly (no driver → a clear `SubstrateUnavailable`, never an `ImportError` — proven by the clean-room smoke). **Live in-engine**: a driver registered at import wires the two engine-owned verbs (`stage_handover`/`advance` → the real impls) and the `stage_handover`/`advance_pipeline` tool calls route through it; `select_unit`/`deliver` (the private dev-loop substrate, which `core/` must not import) and `record_feedback` (the server reconciler inbox) report `SubstrateUnavailable` until wired by their owner. |
| **LessonStore / LessonProvider API** (`ack.lessons`) | **wired + tested** | The curated, versioned (ADR-0004) public delegation seam for scope-partitioned actionable lessons (#601 S14-3 / AD-10) — the stable surface that **unblocks epic #602** (its Distiller registers a provider; lesson semantics live there, not here). `runtime_checkable LessonProvider` (`get_lessons`/`report_lesson`/`brief`) + `set_provider`/`get_provider`; ships in the wheel; imports nothing from the engine. **Fail-soft** with no provider (reads `[]`, writes no-op, a provider error never breaks a turn — lessons are advisory). Scope-priority `brief()` merge (delegated or composed, dedup+limit). **Fail-closed redaction-gated `promote()`** (AD-9): a project-private lesson reaches a broader scope only through an approving redactor. The scope is the engine's opaque `mem_scope` partition string. **Engine-wired** (#601 S14-4): the loop consults the seam at two scope-keyed sites — the handover **read** site appends an advisory lesson `brief` next to the Memory brief, and the task-completion **write** site reports the feedback as a scoped lesson (gated on a registered provider). The write site is **re-homed onto the `post_feedback` Hook-Bus** (#602 2.3/#804): `gx10._lessons_consumer_hook` (registered on provider presence via `_apply_lessons_consumer`) reports on a fresh completion through the real `_advance_pipeline` wrapper, outside the vault lock — one consistent reflection path (no double-report on an already-done re-advance). | **Failure-classification taxonomy** (`ack.failure_class`) | **wired + tested** | The shared, string-valued `FailureClass` enum naming *why* a step failed (epic #602 S602-3) — the single vocabulary the reflection layer reads instead of re-deriving its own. `classify_emission_failure(message, detail)` is a **pure, deterministic, rule-based** mapper over the exact error the validate→reask loop already records (never a model, never raises; conservative default `SCHEMA_INVALID`); the engine `providers.result_failure_class` re-maps the 3-class run taxonomy onto the same enum (one SSOT). `validated_emit` attaches an **additive, advisory** `ValidatedEmitResult.failure_class` on a terminal failure — `None` on success and on any result built without it (**byte-identical** to the pre-#602 shape); it **never** affects control flow on the fail-closed path. Public ACK surface (advisory labels, **not** the contract-SSOT). The field is **produced** on the validate→reask path AND — opt-in per `strategy.enabled` (#602 2.4/#805) — recorded at the **code-agent failover** (`gx10._record_failure_class` → `gx10._last_failure_class()`, surfaced in the `/feedback` response) for the Strategy Revisor consumer. | **Process-Level Self-Correction** (`ack.process`) | **wired + tested** | Correct the *workflow*, not the response (epic #602 S602-6). | **Quality Circuit Breaker** (`ack.quality`) | **wired + tested** | A **separate**, agent-agnostic per-task output-quality breaker (epic #602 S602-9) — distinct from the per-peer availability breaker (`_CODE_AGENT_BREAKER`; folding quality in would corrupt failover). `QualityBreaker` tracks the trend of the mark-only verifier scores (`ack.verify`) and trips on **sustained degradation** (`min_consecutive` scores below `threshold`; an at/above-threshold score resets the streak). **MARK-ONLY**: a trip is advisory — escalate/surface (pause-autoplan opt-in), **never a hard-abort**. **Fail-open-safe** (never raises; a hiccup leaves it untripped). OPT-IN via `quality.enabled` (default OFF → no breaker → no-op byte-identical); `_apply_config` builds/clears the engine's `_QUALITY_BREAKER`, `_quality_breaker()` exposes it. **Wired (#602 2.7 / #808):** a `post_handover` consumer (`gx10._quality_consumer_hook`, registered per `quality.enabled` via `_apply_quality_consumer`; default OFF → no hook → byte-identical) feeds the mark-only Verifier score (`gx10._last_verdict()`) into the breaker on every staged handover and **surfaces** a sustained-degradation trip (`gx10._quality_tripped()` + a `[quality]` advisory line) — **advisory, never a hard-abort**. Public ACK surface (`QualityBreaker`/`QualitySnapshot`). |
| **Verifier / Evaluation** (`ack.verify`) | **wired + tested** | A **mark-only** behavioral-evaluation layer (epic #602 S602-4): `VerdictResult` (passed/score/reason) from three opt-in, transport-injected, secret-free verifiers — `verify_rules` (deterministic predicates), `verify_grounding` (claims grounded by an injected `retrieve`), and the async `verify_with_judge` (LLM-as-judge over the injected `chat`, **budget-gated** via an injected ledger duck-typed on `dispatch.BudgetLedger` — skips when unaffordable, charges only a completed call). **MARK-ONLY**: a verdict can neither relax nor tighten a gate (the fail-closed core is untouched). Every verifier **never raises**; **default-off byte-identical**. **Wired (#602 2.1 / #802):** the engine runs the Verifier on the **dev-task pipeline** — a `pre_handover` Hook-Bus subscriber (`gx10._verifier_hook`, registered per `verify.enabled` via `_apply_verifier`; default OFF → no hook → byte-identical) evaluates each staged task: behavioral rules over `task_json` + (when a memory tier is up) grounding of the handover's claims against the cold store. It stores a **mark-only** `VerdictResult` (`gx10._last_verdict()`) for the Quality breaker (#602 SUB-9 / 2.7) to read — it **never gates** a handover. The opt-in **LLM-judge** (`verify_with_judge`) stays a separate explicit activation (it charges the budget ledger) and is **not** run by this hook; per-`TaskType` eval selection via `LoopProfile.eval_verifiers` (8b) rides this runner. Public ACK surface (`VerdictResult` + the verifiers). |
| **Loop Profiles** (`ack.loop_profile`) | **wired + tested** | Per-`TaskType` loop budgets (epic #602 S602-8a): the **pure** `resolve_loop_profile(...)` deep-merges code defaults ← `loop_profiles.default` ← `loop_profiles.by_type.<type>` into a `LoopProfile` (max_iterations / retry_budget / effort) — only present keys override, `retry_budget` clamped to the hard re-ask ceiling, **never raises**. Schema + defaults in the engine config tree (`loop_profiles`, **empty by default** → falls back to the globals `MAX_ITERATIONS` + re-ask budget → **byte-identical**, single-sourced); `_loop_profile()` (the **default** profile) drives the chat loop's iteration bound. **Seam status (honest):** the **default** profile is live (the chat loop bound), but `by_type` per-`TaskType` **`retry_budget` is now consumed on the dev-task pipeline** (#602 2.6 / #807): `gx10._failover_budget` resolves it per the staged task's type for the code-agent failover escalation (`gx10._revise_on_failure`), layered over `strategy.budget` and clamped to the hard ceiling (a per-type override can only LOWER it). The chat loop's `max_iterations` still uses the default profile (it has no per-task type); `effort`/`eval_verifiers` ride the Verifier (8b). An operator (or the private `conf/loops` override layer — NOT `core/conf`) tunes the default limits; the public default is unchanged. Public ACK surface (`LoopProfile`/`resolve_loop_profile`). **8b wired** (#809): `LoopProfile.eval_verifiers` (the per-type `eval` key) **selects which mark-only verifiers the per-task Verifier runs** — consumed by `gx10._verifier_hook` (empty → the default rules+grounding set; the async LLM-judge stays a separate opt-in consumer). |
| **Strategy Revisor** (`ack.strategy`) | **wired + tested** | The pure failure→action policy (epic #602 S602-7): `revise(failure_class, attempt, budget) → Strategy` (the SSOT) maps a `FailureClass` to a targeted `StrategyAction` (inject-context / narrow-or-clarify / switch-retrieval / ground-then-answer / complete-output / resolve-policy / repair-schema / fail-over / human-escalation) instead of "retry the same way", escalating to a human when the budget is spent. **Pure / deterministic** (no transport, no model, never raises). Two **opt-in** application seams: `validated_emit`'s optional `strategist` param appends the strategy hint to the re-ask turn (**byte-identical when not passed**; a strategist error is swallowed), and the engine `providers.code_agent_strategy` maps a code-agent run result through the same SSOT. **Wired (#602 2.5 / #806):** the engine consults the Strategy Revisor on the **code-agent failover** — `gx10._revise_on_failure` runs `providers.code_agent_strategy(result, attempt, budget)` per task on the `/feedback` failure path and **surfaces a `HUMAN_ESCALATION`** when the per-task attempt budget (`strategy.budget`) is spent, instead of an endless silent failover (`gx10._last_strategy()`; a successful run resets the counter). OPT-IN per `strategy.enabled` (default OFF → byte-identical); **MARK-ONLY** (it surfaces, never adds a new hard-abort). The richer re-ask actions (inject-context / switch-retrieval) ride `validated_emit`'s `strategist` in the MPR plugin, not the code-agent failover. Public ACK surface (`Strategy`/`StrategyAction`/`revise`). |
| **Loop-Intelligence Hook-Bus** (`ack.hooks`) | **wired + tested** | The keystone seam (#602 2.0 / #690) the reflection consumers subscribe to instead of hard-wiring call-sites into `gx10`: a standalone, dependency-inverted, fail-soft event bus over the agent-loop boundaries. The engine **publishes** 7 canonical events — `pre_turn`/`post_generate`/`post_toolresult` from `run()`, and `pre_handover`/`post_handover`/`pre_advance`/`post_feedback` from the `_stage_handover`/`_advance_pipeline` wrappers (outside the `_vault_lock`). `register_hook` is additive + idempotent (fail-loud on an unknown event / non-callable); `dispatch` is **observer-only** (a hook may ABORT, never relax a gate), **fail-soft** (a bad subscriber never breaks a turn), cancel/budget-aware, copy-on-write + snapshot for the multi-threaded engine, and an **O(1) no-op when no hook is registered → byte-identical default**. ## Memory

The engine has long-term memory **wired in**: a `query_memory` tool, store-on-task-
completion, and stage-time context injection, backed by `engine/memory.py` — a small,
secret-free client for a **Mem0-style HTTP service** (`POST /add`, `POST /search` with
`graph=false`, `GET /health`). Store + search are verified live against the reference
Mem0 stack (Qdrant + Neo4j + BGE-M3, LLM pointed at the local model).

**Three tiers (all server-side, fail-soft).** *Hot* = the model window (a bounded working
set, **token-accurately** trimmed against the model wall — real token counts from the served
model's tokenizer, with a calibrated chars/token fallback). *Warm* = `engine/warm.py`, a BSD-licensed in-memory store
(`GX10_WARM_URL`, e.g. Valkey) holding the rolling conversation summary + recent-turn state — which
**survive a restart** and are **shared across the reasoning workers** — plus a separate short-TTL
(~3 min) retrieval cache-aside. The shipped compose defaults `GX10_WARM_URL` to the `mem-valkey`
loopback, so the warm tier wires automatically under `--profile memory` (fail-soft otherwise). *Cold* = the Mem0 vector(+graph) store below. On top of the tiers:
rolling/hierarchical summarization on eviction (raw archived to cold first, then a **bounded
tail-first** summary input so the summarizer can't itself overflow the window, prefix-stable),
per-turn RAG assembly (warm cache-aside, the RAG block itself token-budgeted), token-accurate
budgeting (real token counts via the served model's tokenizer — the vLLM `/tokenize` endpoint —
with a calibrated chars/token fallback; the model window is auto-adopted from `/v1/models` at boot)
that scales the working set to the model window without overflowing it, a **pre-flight overflow guard** (before each call, reserve output —
`generation.max_tokens` / `GX10_MAX_TOKENS`, default 8192 — + the tools schema + the conditional
thinking budget; one emergency whole-round trim, then a clear `ContextOverflowError` instead of a raw
vLLM 400), chunked + recency-ranked long-artifact storage, an on-demand
`deep_query_memory` graph tool, and worker memory (workers read the shared summary +
per-item retrieval, a single-writer reducer consolidates their writes). Each piece is
individually flag-gated (default-on where a memory/warm service is configured); with the
warm/cold tiers down a turn still completes. The hot read path stays vector-only and off the
timeout-prone graph path.

**This repo ships the wiring, never any memory content.** Memory is a runtime service,
not data in the codebase:

- **Off by default.** With no `GX10_MEMORY_URL` (and no `conf/memory/memory.json`) the
  `MemoryManager` is never constructed → all hooks stay inert and no tool is offered.
- **Bring your own service _and_ corpus.** Point the engine at your Mem0 endpoint
  (`GX10_MEMORY_URL=http://your-mem-host:8800`, optional `GX10_MEMORY_AGENT`). The store
  starts **empty** — Ironclad accumulates its own memory from task completions; any
  pre-existing corpus is yours to import, into your own namespace, and never lives here.
- **Namespace.** Reads/writes use `agent_id` (default `ironclad`) and an optional
  `user_id`; set them to match wherever your content lives.

So the public artifact is **wired but empty** — the integration works, the content is
the operator's.

## Extension & self-maintenance — what's done vs in development

Ironclad is meant to be **extended and maintained through itself**, with a clean DEV/Prod
split. Honest split of that capability:

- **Done (wired + tested) — the user-facing surface.** The **open plugin API**
  (`GX10_PLUGINS_DIR`, [`plugin-api.md`](plugin-api.md)), the **pluggable code-agent CLI**
  (`GX10_AGENT_CMD`, [`code-agents.md`](code-agents.md)), the **dev container** build+test
  gate ([`dev-environment.md`](dev-environment.md)), the runtime `GET /doctor` self-check,
  and the beginner **self-maintenance guide** ([`self-maintenance.md`](self-maintenance.md))
  all ship and are covered by tests (`test_plugins.py`, `test_client_pool.py`,
  `test_doctor_endpoint.py`). The extension model is *additive*: plugins dock onto a stable
  contract, the core is never *forked* — plugins dock through the stable contract and `core/` stays standalone + secret-free (boundary-check enforced; the check guards `core/`, it does not sandbox plugin code).
- **Mostly done — our internal release machinery.** The **DEV → Prod → Public promote** now
  runs as a **single fail-closed gated command**: boundary (core + client) → the full test suite
  (with a JUnit proof) → docs gate (CHANGELOG + doc-reality-audit) → export gates → a
  **PRE-publish clean-room** (wheel → fresh venv → import-smoke → an example plugin builds against
  the installed SDK) → a 2nd-instance review → publish → prod redeploy. It is **dry by default**
  and stops before anything irreversible. The remaining step is **full automation** (a scheduled
  evening sync). Downstream users never touch any of this — they pull the framework and extend it
  over the plugin API, or fork it freely (Apache-2.0).

So the way you *extend* Ironclad is shipped and tested; the way *we harden our own releases* is a
single gated flow, with scheduled automation the last step.

## Reference load test

Run against the full reference stack on the DGX Spark (2026-06-17): all five
containers up — `vllm-35b`, `mem-api`, `mem-qdrant`, `mem-neo4j`,
`ironclad-orchestrator` (all healthy) — driven from a workstation over the LAN.
*(The warm tier `mem-valkey` was added after this run — `--profile memory` now brings up **five**
containers: the four `mem-*` services + the always-on orchestrator; `vllm` is a separate
`--profile model`.)*

**Fan-out (8 independent reasoning prompts, concurrent):**

| metric | value |
|--------|-------|
| success | 8 / 8 |
| wall-clock | 1.2 s |
| Σ per-prompt latency | 7.1 s |
| **speedup vs serial** | **5.8×** |
| aggregate throughput | ~118 tok/s |

**Chat (3 sequential turns, single agent — serialized by design):**

| metric | value |
|--------|-------|
| mean turn latency | 2.1 s |
| decode rate | ~55–68 tok/s |
| answers | all correct, in the configured language (German) |

Interpretation: the parallel reasoning path scales near-linearly to the model's batch
width (`max_num_seqs=8`); the conversational path is intentionally one-turn-at-a-time
behind the agent lock. Both are healthy on a single GB10.

## Sealed channel — live verification (Phase d)

Verified on the reference GPU (2026-06-17) against a server in the `sealed` profile,
driven from a workstation over a **real SSH local-forward** (the client-managed tunnel),
without touching the production orchestrator:

| check | result |
|-------|--------|
| server bind | `127.0.0.1:8101` only (loopback — never the LAN) |
| `/health` | `sealed: true`, `security.profile: sealed` |
| `/tasks`, `/session/open` without the secret | **401** |
| `/session/open` with the secret → `/tasks` with secret + session | **200** |
| real model turn (`/chat`) through the sealed tunnel | answered correctly |
| after `/session/close` → `/health`, then `/tasks` with the stale session | `sealed: true`, **401** |

The channel is open exactly while a live session exists and seals the moment it ends —
OS-enforced when the tunnel closes, app-enforced by the session heartbeat. Single-tenant
throughout: the secret authenticates the deployment, not a user.
