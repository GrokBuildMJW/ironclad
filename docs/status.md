# Project status & wiring

> ## ⚠️ Read this first
>
> Ironclad's engine grew out of a **proven, in-production orchestrator** (years of
> daily use). It is currently undergoing a **complete redesign**: the original
> single-process CLI is being split into a headless **server** + thin **client**,
> containerized, and given a reasoning-worker fan-out and a purpose-built TypeScript terminal client.
>
> **Not everything is re-wired or re-tested yet.** The pieces below are marked
> honestly: *proven* (inherited, battle-tested), *wired + tested* (rebuilt and
> verified in the new architecture), *placeholder* (hook exists, backend/logic not
> yet ported), or *opt-in* (off by default). Treat `main` as a development snapshot.

This document is the single source of truth for **what actually works right now**.

Switch cleanup: `setup.type` is the sole provider-topology authority; legacy
`providers.enabled` / `GX10_PROVIDERS` inputs and unwired `providers.scoring.*` values are warning-only
tombstones. ACE and its PlaybookStore provider stay always on, with legacy lesson files used only as migration
input; `lessons.enabled`, `lessons.max_per_scope`, and `safety.constraint_conflict_detect` are tombstones.
`process.hints_enabled` controls only the ACE-backed pre-turn hint, while `process.enabled` is a one-release
alias. `/auto on|off` alone owns watcher runtime state; `watcher.enabled` is retired. Live `quality.threshold`,
`quality.min_consecutive`, and `quality.window` changes atomically rebuild the breaker, retain bounded score
history, preserve its live latched/recovered state, and trip an unlatched rebuild only for a sustained trailing
low-score streak. `framing_notes.enabled` replaces `constraint_gate.enabled` (one-release alias)
and controls only `record_constraints` exposure/capture—not design authority or implementation gating.

Constraint/framing note: the former product constraint gate, typed HARD-floor reader, product
constraint-fork ledger, `/approve constraint`, `/dismiss constraint`, and constraint-envelope ACE leg are
retired. `record_constraints` is now optional framing-note capture under `notes/framing.md`; it can be
exposed by `framing_notes.enabled`, but it does not block design recording, decomposition, implementation
handovers, or approval. The design lifecycle is always on: implementation tasks compare their typed `language`
to the approved `decisions/design.md` metadata, and coder
handovers inject that approved design standard plus any `## Build policy` section. The always-on egress boundary
reads `network`, `allow`, and `deny` only from that section; no section means the design is non-build-enforcing,
while a present section must declare exactly `network: none|declared|open`. This policy is not part of the typed
build hard-check, and `_design_build_check` deliberately never compares `network`. The shared architecture-fork
worker and `/fork` proposal view remain available; only the product constraint-envelope leg was removed.
When a design choice is genuinely open, `/design --options [N]` is the explicit operator-triggered path for
requesting N comparable proposal variants with `## Trade-offs`; N is constrained to 2..8 and defaults to 2.
The command reports how many proposal files were actually recorded and warns if the model records fewer
than requested.
The engine never auto-fans-out from prose; `/design --options [N]` is always available for an active unit.

## Rust egress limitations

The Rust egress layer is deliberately a build-boundary tripwire, not a runtime confinement promise. The
hardened Rust probe neutralizes Cargo/Rust environment influence and refuses to run Cargo on a project whose `.cargo/config`
contains `rustc-wrapper`, `[env]`, `[source]`, `rustflags`, `credential-provider`, target linker/runner
overrides, `include`, or a path-based toolchain, or whose `.cargo` directory is a symlink. That fail-closed
gate prevents the probe from executing under project-supplied Cargo indirection, but it also means such
projects receive a refusal/advisory result instead of a probe verdict.

Remaining limits are explicit: a `build.rs` that catches and swallows a failed network connect can be a false
negative for the hermetic probe; regex-based `.rs` scanning can report false positives in comments or string
literals; low-level crates such as `socket2`, `mio`, `nix`, and `libc` are awareness-only below the
`std::net` surface; transitive Rust dependency egress requires a checked-in `Cargo.lock`; and Node ecosystem
analysis is not implemented, so the always-on gate currently dispatches only the Python and Rust analyzers.

> **Release model.** Ironclad is **pre-release** (`0.0.x`, alpha). Tagged releases are
> published on **PyPI** as `ironclad-ai` and as **GitHub Releases** (latest `v0.0.30`);
> the importable wheel is the Agent-Contract-Kernel (`ack` + `ack.lodestar`), while the
> orchestration engine ships as runnable scripts until its API stabilizes. There is **no
> stable/1.0 release yet** — APIs, layout, and config may change between `0.0.x` versions.

### Design-document compatibility

Legacy design migration is lazy and project/track scoped. It runs under the vault lock before the first design
status, read, record, or approval for a unit:

| `decisions/design.md` state | Gate behaviour | Normalisation |
|-----------------------------|----------------|---------------|
| No legacy decision | Existing proposals remain unchanged. | No migration. |
| `approved: true` | The decision remains authoritative and byte-identical. | No rewrite or invented proposal. |
| `approved: false` | Implementation remains blocked. | Atomically moved to the next collision-free proposal and normalized to `type: proposal`, `approved: false`. |
| Missing, malformed, oversized, or unreadable | Every design operation refuses with an actionable error. | Bytes remain unchanged; the document is never overwritten, deleted, or treated as approved. |

Migration and promotion use temp-write, file fsync, atomic replace, and directory fsync where supported. No
eager whole-vault scan occurs at boot.

### Always-on, non-destructive design variants

Every recorded design is a **non-destructive variant**, not an in-place
overwrite: `record_design` writes a retained proposal to `proposals/design-<n>.md` (numbered from 1) and
never touches an existing approved decision — exploring an alternative approach preserves the prior one, so
the operator can compare variants and pick. `/approve design [<id>]` **promotes** the chosen variant
(`<id>` = `2` or `design-2`; the sole proposal when omitted) atomically into `decisions/design.md` as the
decision (`type: decision`, `approved: true`). The approved decision **carries the decided standard**: the
typed `language` metadata plus any `## Build policy` prose section (dependency/egress guidance the build
honours) is preserved verbatim on promote. Inside that section, optional structured egress lines are
machine-readable for always-on egress enforcement: exactly one valid posture
`network: none|declared|open`, plus optional `allow: pkgA, pkgB` and `deny: pkgC`. A design with no
`## Build policy` is non-build-enforcing and advances; a present section with a missing or invalid posture
refuses advance, and an unreadable approved design vault refuses reconciliation. `decisions/`
purity holds — only the approved decision lives
there; proposal variants stay under `proposals/` (retained after promotion as provenance). The promoted
proposal is stamped `promoted: true` and linked back to
`decisions/design.md` via `promoted_to`; the decision declares the proposal's full relative path in a typed
`supersedes` edge, which reconcile resolves into `GRAPH.json`. Because recording is non-destructive,
restating a design after approval does not auto-revoke the decision; the build stays on
the approved decision until the operator promotes the new variant (`/approve design <id>`; re-running
`/approve` names the newer proposals).
At the build boundary, implementation-unit `stage_handover` (by existing `task_id`) and `plan_units` compare
the task typed `language` to the approved decision's `language` and refuse drift before writing. The coder
handover injects that approved typed language plus the `## Build policy` section as authoritative guidance;
an approved design with neither field is a no-op for this build check/injection. The `network`/`allow`/`deny`
egress policy is not part of the typed hard-check (`_design_build_check` never compares `network`), but it
is enforced at advance time without a config bypass (see the status row above). When
`framing_notes.enabled` is on, framing notes remain
capture-only context; the coder handover injects the
approved design standard, not the framing notes.

`design_gate.enabled` is a deprecated tombstone: either legacy value emits one warning, is ignored, and cannot
be set at runtime. It cannot restore the destructive single-document overwrite path.

## Memory

The engine has long-term memory **wired in**: a `query_memory` tool, store-on-task-
completion, and stage-time context injection, backed by `engine/memory.py` — a small,
secret-free client for a **Mem0-style HTTP service** (`POST /add`, `POST /search` with
`graph=false`, `GET /health`). Store + search are verified live against the reference
Mem0 stack (Qdrant + Neo4j + BGE-M3, LLM pointed at the local model).

**Three tiers (all server-side, fail-soft).** *Hot* = the model window (a bounded working
set, **token-accurately** trimmed against the model wall — real token counts from the served
model's tokenizer, with a calibrated chars/token fallback). *Warm* = `engine/warm.py`, a BSD-licensed in-memory store
(`GX10_WARM_URL`, e.g. Valkey) holding the rolling conversation summary + recent-turn state — which
**survive a restart** and are **shared across the reasoning workers** — plus a separate short-TTL
(~3 min) retrieval cache-aside. The shipped compose defaults `GX10_WARM_URL` to the `mem-valkey`
loopback, so the warm tier wires automatically under `--profile memory` (fail-soft otherwise). *Cold* = the Mem0 vector(+graph) store below. On top of the tiers:
rolling/hierarchical summarization on eviction (raw archived to cold first, then a **bounded
tail-first** summary input so the summarizer can't itself overflow the window, prefix-stable, and **query-aware** — the summary biases retention toward the current task, fail-soft to a generic instruction when no turn is in scope; a **default-off** proactive accountant sheds the oldest rounds early via summary once ingestion crosses a soft mark, bounded by a shared per-turn summarize rate-limit),
per-turn RAG assembly (warm cache-aside, the RAG block itself token-budgeted), token-accurate
budgeting (real token counts via the served model's tokenizer — the vLLM `/tokenize` endpoint —
with a calibrated chars/token fallback; the model window is auto-adopted from `/v1/models` at boot)
that scales the working set to the model window without overflowing it, a **single ingestion choke-point cap** (every `read_file`/`list_directory`/`search_files`/`execute_command` result — and the local-tool bridge return — capped to the live per-turn budget at one place, so no one tool result can overflow the window; structured/already-budgeted payloads like `web_search`/`parallel_reason`/MPR/memory skip this destructive cap but still cross the mandatory injection fence), plus **pre-allocation filesystem bounds**: Python and Ink `read_file` refuse files above 16 MiB, `list_directory` retains at most the 200-entry cap plus one overflow probe and reports “many” instead of materializing a huge directory, and `search_files` lazily stops at 50 hits or 1,000 candidate files while reading at most 1 MiB per file and marking every truncated result. The server-side `edit_file` read-modify-write path, model-facing `review(paths=...)` reader, and user-project vault document scanner share the 16 MiB capped full-text reader; `edit_file` has no Ink twin. **Ranged/pattern `read_file`** (`start`/`end` line numbers or a regex `pattern` window, mirrored in the ink client, so the model reads only the relevant slice of a bounded file instead of the whole thing; a bad range falls back to the head+tail cap), a **pre-flight overflow guard** (before each call, reserve output —
`generation.max_tokens` / `GX10_MAX_TOKENS`, default 8192 — + the tools schema + the conditional
thinking budget; the output reserve is an **adaptive ceiling**, not a fixed floor: when the full reserve
would not fit, it reserves **less** output — down to `context.min_output_tokens` / `GX10_MIN_OUTPUT_TOKENS`
(default 1024) — so a marginal-overflow turn proceeds **losslessly** with a shorter answer instead of
failing; a `context.overflow_safety_tokens` / `GX10_OVERFLOW_SAFETY` headroom (default 1536) is kept below
the wall so the clamp never targets it to the token — absorbing the gap between the engine's estimate and
vLLM's exact rendered-prompt count (template + tools serialization); only when even a minimal answer will
not fit does it emergency whole-round trim, then **iteratively truncate the largest turns' content** — so an
agentic loop (one user turn, many accumulated tool reads, no round boundary to evict) degrades to head+tail
excerpts rather than dying — and the discarded slice is **cold-archived** (`fragment_trim`) so RAG can re-inject it query-aware next turn (an optional **default-off** `emergency_summarize` replaces the raw drop with a bounded summary under a hard timeout, fail-through to raw) — then, last, a clear `ContextOverflowError` instead of a raw vLLM 400),
chunked + recency-ranked long-artifact storage, an on-demand
`deep_query_memory` graph tool, and worker memory (workers read the shared summary +
per-item retrieval, a single-writer reducer consolidates their writes). Each piece is
individually flag-gated (default-on where a memory/warm service is configured); with the
warm/cold tiers down a turn still completes. The hot read path stays vector-only and off the
timeout-prone graph path.

**This repo ships the wiring, never any memory content.** Memory is a runtime service,
not data in the codebase:

- **Off by default.** With no `GX10_MEMORY_URL` (and no private memory config file) the
  `MemoryManager` is never constructed → all hooks stay inert and no tool is offered.
- **Bring your own service _and_ corpus.** Point the engine at your Mem0 endpoint
  (`GX10_MEMORY_URL=http://your-mem-host:8800`, optional `GX10_MEMORY_AGENT`). The store
  starts **empty** — Ironclad accumulates its own memory from task completions; any
  pre-existing corpus is yours to import, into your own namespace, and never lives here.
- **Namespace.** Reads/writes use `agent_id` (default `ironclad`) and an optional
  `user_id`; set them to match wherever your content lives.

So the public artifact is **wired but empty** — the integration works, the content is
the operator's.

## Extension & self-maintenance — what's done vs in development

Ironclad is meant to be **extended and maintained through itself**, with a clean DEV/Prod
split. Honest split of that capability:

- **Done (wired + tested) — the user-facing surface.** The **open plugin API**
  (`GX10_PLUGINS_DIR`, [`plugin-api.md`](plugin-api.md)), the **pluggable code-agent CLI**
  (`GX10_AGENT_CMD`, [`code-agents.md`](code-agents.md)), the **dev container** build+test
  gate ([`dev-environment.md`](dev-environment.md)), the runtime `GET /doctor` self-check,
  and the beginner **self-maintenance guide** ([`self-maintenance.md`](self-maintenance.md))
  all ship and are covered by tests (`test_plugins.py`, `test_client_pool.py`,
  `test_doctor_endpoint.py`). The extension model is *additive*: plugins dock onto a stable
  contract, the core is never *forked* — plugins dock through the stable contract and `core/` stays standalone + secret-free (boundary-check enforced; the check guards `core/`, it does not sandbox plugin code).
- **Mostly done — our internal release machinery.** The **DEV → Prod → Public promote** now
  runs as a **single fail-closed gated command**: boundary (core + client) → the full test suite
  (with a JUnit proof) → docs gate (CHANGELOG + doc-reality-audit) → export gates → a
  **PRE-publish clean-room** (wheel → fresh venv → import-smoke → an example plugin builds against
  the installed SDK) → a 2nd-instance review → publish → prod redeploy. It is **dry by default**
  and stops before anything irreversible. The remaining step is **full automation** (a scheduled
  evening sync). Downstream users never touch any of this — they pull the framework and extend it
  over the plugin API, or fork it freely (Apache-2.0).

So the way you *extend* Ironclad is shipped and tested; the way *we harden our own releases* is a
single gated flow, with scheduled automation the last step.

## Reference load test

Run against the full reference stack on the DGX Spark (2026-06-17): all five
containers up — `vllm-35b`, `mem-api`, `mem-qdrant`, `mem-neo4j`,
`ironclad-orchestrator` (all healthy) — driven from a workstation over the LAN.
*(The warm tier `mem-valkey` was added after this run — `--profile memory` now brings up **five**
containers: the four `mem-*` services + the always-on orchestrator; `vllm` is a separate
`--profile model`.)*

**Fan-out (8 independent reasoning prompts, concurrent):**

| metric | value |
|--------|-------|
| success | 8 / 8 |
| wall-clock | 1.2 s |
| Σ per-prompt latency | 7.1 s |
| **speedup vs serial** | **5.8×** |
| aggregate throughput | ~118 tok/s |

**Chat (3 sequential turns, single agent — serialized by design):**

| metric | value |
|--------|-------|
| mean turn latency | 2.1 s |
| decode rate | ~55–68 tok/s |
| answers | all correct, in the configured language (German) |

Interpretation: the parallel reasoning path scales near-linearly to the model's batch
width (`max_num_seqs=8`); the conversational path is intentionally one-turn-at-a-time
behind the agent lock. Both are healthy on a single GB10.

## Sealed channel — live verification

Verified on the reference GPU (2026-06-17) against a server in the `sealed` profile,
driven from a workstation over a **real SSH local-forward** (the client-managed tunnel),
without touching the production orchestrator:

| check | result |
|-------|--------|
| server bind | `127.0.0.1:8101` only (loopback — never the LAN) |
| `/health` | `sealed: true`, `security.profile: sealed` |
| `/tasks`, `/session/open` without the secret | **401** |
| `/session/open` with the secret → `/tasks` with secret + session | **200** |
| real model turn (`/chat`) through the sealed tunnel | answered correctly |
| after `/session/close` → `/health`, then `/tasks` with the stale session | `sealed: true`, **401** |

The channel is open exactly while a live session exists and seals the moment it ends —
OS-enforced when the tunnel closes, app-enforced by the session heartbeat. Single-tenant
throughout: the secret authenticates the deployment, not a user.
